**为什么今天推荐这篇：** 今天的简报中，Agent 基础设施标准化、端侧推理和成本管控是主旋律。但多数报道停留在“高层趋势”层面，而这篇博客提供了一个真实产品的架构演进路线图：从简单调用 LLM 到构建 Agent + Skill 体系。它触及了每个 Agent 开发者都会遇到的实际问题——成本、易用性、拆分质量。在 Hugging Face CEO 强调开放模型、Meta 讨论 token 预算的背景下，BaoCut 的决策过程（为何放弃纯 LLM 方案，如何设计 Skill 层）对 2027 年 Agent 产品开发具有直接参考价值。

- **核心观点一：** 纯 LLM 方案在长视频字幕翻译中成本高、易用性差（需要用户配置 API Key）、编辑效率低。将任务拆解为多个 Skill（语音识别、翻译、对齐、字幕渲染）后，每个 Skill 由专门的模型或工具执行，而非全部依赖一个 LLM。
- **核心观点二：** Agent + Skill 架构的核心收益是成本可预测：每个 Skill 有独立的定价和缓存策略，避免一次调用消耗大量 token。同时，用户可以逐个 Skill 进行人工校正，而非面对一整段 LLM 输出。
- **核心观点三：** 架构中的“Skill 编排”是关键挑战：如何定义 Skill 之间的数据接口？如何保证粒度和复用性？BaoCut 通过定义统一的输入/输出 Schema 和错误恢复策略，实现了 Skill 的可插拔。

适合：Agent 产品经理、Harness 设计者、正在从 LLM 封装转向 Agent 架构的开发者 · 阅读时间：约 10 分钟
